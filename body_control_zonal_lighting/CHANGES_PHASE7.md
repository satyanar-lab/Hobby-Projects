# Phase 7 — Demo Polish

## Overview

Phase 7 closes the loop between the NUCLEO-H753ZI rear node and the Linux
HMI: `LampStatus` events generated by the STM32 firmware now flow all the
way back through the Central Zone Controller's operator service to the
terminal HMI, making the hardware demo fully bidirectional.  The phase also
ships the hardware walkthrough document and verifies the end-to-end path
with real LEDs.

---

## Changes

### End-to-end status event path

With Phase 6 complete, the STM32 `RearLightingFunctionManager` called
`ApplyCommand()` and updated its internal state, but the resulting
`LampStatus` event was not always reaching the operator clients.  Phase 7
confirms and hardens the full event flow:

```
NUCLEO firmware
  RearLightingFunctionManager::ApplyCommand()
    → RearLightingServiceProvider::OnCommandApplied()
    → LwipUdpTransportAdapter::SendEvent()
    → UDP → 192.168.0.10:41000

Linux CZC
  UdpTransportAdapter::ReceiverLoop()
    → RearLightingServiceConsumer::OnTransportMessageReceived()
    → CentralZoneController::OnLampStatusReceived()
    → LampStateManager::UpdateLampStatus()
    → OperatorServiceProvider::OnLampStatusReceived()  (status observer)
    → UdpTransportAdapter::SendEvent() → UDP → :41003

hmi_control_panel_terminal / hmi_control_panel_qt
  OperatorServiceConsumer::OnTransportMessageReceived()
    → MainWindow::OnLampStatusUpdated()
    → HmiViewModel update + display refresh
```

### `app/central_zone_controller/main.cpp`

Added a `FanoutTransportAdapter` that forwards every outgoing rear-lighting
command to both the vsomeip/UDP simulator path and the `DirectUdpTransportAdapter`
pointed at the NUCLEO (`192.168.0.20:41001`).  The fanout adapter ensures
the Linux simulator and the real hardware can run concurrently without
modification to any service or application layer code.

### `src/transport/ethernet/direct_udp_transport_adapter.cpp`

Minor fix: the `is_initialized_` flag is now set before the availability
notification is fired so a handler registered in `SetMessageHandler` after
`Initialize()` still receives the replay correctly.

### `doc/demo_walkthrough_hardware.md` *(new)*

Comprehensive step-by-step hardware bring-up guide covering:

- Hardware setup: NUCLEO-H753ZI, Ethernet cable, LED wiring table (PB1–PB5).
- Network configuration: static IPs, `sudo ip addr` command, interface rename.
- Build instructions for both the STM32 firmware and the Linux applications.
- Flash procedure via STM32CubeProgrammer.
- Four-terminal run sequence with expected PuTTY output at each step.
- Expected LED and serial behavior for every HMI button.
- Architecture section explaining zonal SDV design versus traditional BCM.

### Engineering comments pass

All source files received `/** */` block comments on every class and
public function, inline `//` comments on every non-obvious code block,
`///< ` trailing comments on every enum value, and field comments on every
domain struct.  No logic was changed; only documentation was added.

## Verified behavior (hardware)

| Action | NUCLEO LED | PuTTY | HMI (terminal) |
|---|---|---|---|
| Toggle left indicator | PB5 blinks 500 ms | `[INF] Lamp command applied: ON` | LeftIndicator → On |
| Toggle hazard | PB2+PB5+PB1 blink in phase | `[INF] Hazard: ON` | HazardLamp → On |
| Toggle park lamp | PB3 steady | `[INF] Lamp command applied: ON` | ParkLamp → On |
| Left indicator while hazard active | no change | `[WRN] Indicator blocked: hazard active` | no change |
| Toggle hazard off | PB2+PB5+PB1 off | `[INF] Hazard: OFF` | HazardLamp → Off |

All nine GoogleTest cases remain green.  No domain types, codec, or
service interface contracts were modified.  The `rear_lighting_node_simulator`
